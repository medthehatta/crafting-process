# Introduction

Ultimately this repository understands "processes" that take outputs from one process and pass it to inputs of another process.  We assemble graphs of these processes to produce desired outputs and we try to find the number of repeats of each process required to get the desired output with as little waste (as few processes and as little mismatch between outputs and inputs) as possible.

# crafting_process/solver.py
The code for solving a mixed integer linear programming problem expressed as a matrix.  

The problem being solved is M x = b where b is a vector of "leak" in a connection between two processes (i.e. each component of b is the mismatch between input to and output from that edge of the ith kind of resource).  In general the closer to the zero vector we require b to be, the more processes we need to find a good match between input and output rates (because it will give our ratios more granularity).  So we have `best_milp_sequence` which solves the mixed integer linear programming (milp) problem with b increasingly smaller, so the initial values are more wasteful of resources but require smaller process counts, and as b shrinks the resources are better "impedance-matched" between the processes, but we require more such processes.

# crafting_process/process.py
This houses the `Process` class which represents these fundamental operations that take inputs and produce outputs.  It is fairly thin and mostly has convenience functions for interacting with these process objects.  In particular these can be assembled into graphs of processes (see graph.py).

# crafting_process/library.py
The way this repo is used, it requires the user specifying a big list of processes in a small domain specific language.  This module knows how to build Process objects from these user-provided textual definitions and how to store them in a repository that can be searched.  It also provides functionality for doing searches for processes through the repository.  The repository is called a `ProcessLibrary`.  Most of the parsing is performed by the sorta poorly named `_parse_process_header`.  An example input file make look like this (showcasing a few different syntaxes):

```
some output | process_name: duration=1
2 some input + 3 another input

another output | different_process: = another input + 6 input3

foo = 2 bar

output1 + 2 output2 = 10 single input

```

# crafting_process/graph.py
The code here represents graphs of processes connected to each other.  The class `GraphBuilder` is a bit of a misnomer as this is used to represent the graphs themselves, as well as operations for joining graphs to each other.

It introduces a few concepts, namely that of "kind" which is the resource type being passed along graph edges between subgraphs (and ultimately between their constituent processes).  And that of a "pool" which represents a source and sink for a given resource that lies along edges between processes.  Ultimately when building the matrix for solving the milp problem, the edges are actually represented by these pools, which have input and outputs (or input and output rates for non-batch processes).

The GraphBuilder has the methods `build_matrix()` and `build_batch_matrix()` which produce the matrices used for milp solving.  The difference between them is that `build_matrix()` assumes the processes are all continuous (in the process sense, i.e. asynchronous and have durations), so the flow _rates_ are used along the graph edges; whereas `build_batch_matrix()` assumes the processes must be done in batches and cannot be run simultaneously, so the durations ultimately are not meaningful, and absolute resource _counts_ are passed along the graph edges.

This distinction is a little bit gross and I would like to eventually make it so batch and continuous processes can coexist in the same graphs, but I have not figured out how to do that yet.

# crafting_process/orchestration.py
This module knows how to build graphs of processes when an output is requested.  It looks through the library and iteratively tries to find processes that produce the desired outputs, and for those processes finds processes that can feed the inputs.  Building these graphs is expensive: it is very possible for there to be multiple ways to produce a given kind of resource, so this will build multiple graphs, one for almost every possible way to produce any constituent input.

It is a bit more constrained in that it does not find one input at a time, but finds inputs that can satisfy an entire "layer" of processes: i.e. if A requires resource kinds from B and C, it will find B and C to satisfy as many inputs to A as possible; then it will try to find a combination of processes to satisfy as many inputs to both B and C as possible; etc.  This converges better than trying to find processes that produce one resource at a time beacuse if some process Q produces more than one output (q1 and q2 say) and some processes P1 and P2 each require only q1 or q2, you can get a situation where you have to repeat Q multiple times, and where you have to "resolve" the inputs to Q multiple times, and you can wind up with deep nested loops that are numerous and get way out of hand.

# crafting_process/ops.py
The code in here used to be similar to the code in orchestration.py, but it was intended to be a "API-ready" interface to the functions.  I think this is now unused by all client code as it quickly got too unweildy.  I am using orchestration.py instead of this in every case I can, which at this point I think is all of the use cases I have.

# crafting_process/augment.py
This module is not well-integrated into the code base at this time, but this provides mechanisms for deriving new processes from old processes and automatically enriching the ProcessLibrary with these derived processes.  An example of this would be if it's possible to "upgrade" a process so it operates more quickly or with greater yield.
